## 명사(Noun)로서의 Map: "만능 저장소"

데이터를 저장하고 관리하는 자료구조(Data Structure)로서의 Map이다.

<br>

**정체성 및 구현**

Map은 Key(키)와 Value(값)의 쌍으로 이루어진 데이터 저장소이다.
각 언어별로 다음과 같이 다양하게 불린다.
* C#: `Dictionary<TKey, TValue>`
* Java: `HashMap`
* C++: `std::map` (트리 기반), `std::unordered_map` (해시 기반)
* JavaScript: `Map`, `Object`



**메모리 생태학 (객체여야 하는 이유)**  
Map은 구조체(Value Type)가 아닌 객체(Reference Type)로 구현된다.
* **가변성(Variability):** 데이터가 0개일 수도, 100만 개일 수도 있는 유동적인 크기를 가지므로 공간이 넉넉한 힙(Heap) 메모리에 저장되어야 한다.
* **복사 비용(Copy Cost):** 구조체라면 함수 전달 시 데이터를 통째로 복사(Deep Copy)해야 하므로 심각한 성능 저하가 발생한다. 객체로 구현되어야 주소(참조)만 가볍게 넘길 수 있다.

**객체(Class) vs 맵(Map) 선택 기준**

| 구분 | 객체 (Class) | 맵 (Map) |
|---|---|---|
| **비유** | 아파트 (설계도대로 지음) | 물류 창고 (유연함) |
| **특징** | 구조가 고정되어 있고 변수명이 문자열로 제한됨 | 실행 중에 자유롭게 키를 추가/삭제할 수 있고 모든 타입이 키가 됨 |
| **용도** | 데이터의 구조(Structure)가 확실할 때 | 유연한 검색(Search)과 관리가 중요할 때 |

**자바스크립트(JS)에서의 대혼란과 정립**  
자바스크립트에서는 일반 객체(`{}`)와 전용 `Map`이 공존한다.  
일반 객체는 키가 무조건 문자열로 강제되며, 프로토타입 상속으로 인한 오염 위험이 있다.  
또한 크기(`length`) 계산 시 $O(N)$이 소요되고, `for...in` 순회 시 부모 속성까지 조회될 위험이 있다.  
반면 ES6부터 도입된 `Map`은 모든 타입이 키가 될 수 있고 상속 없는 깨끗한 빈 방을 제공한다.  
`size` 속성으로 $O(1)$ 크기 확인이 가능하며 안전한 순회를 보장한다.

## 동사(Verb)로서의 Map: "변환 공장"

데이터의 흐름을 변형시키는 고계 함수(Higher-Order Function)로서의 Map이다.

<br>

**정체성 및 언어별 스타일**  
리스트 내 모든 요소를 특정 변환 함수($A \rightarrow B$)에 통과시켜, 원본 데이터를 변경하지 않고(Immutable) 새로운 리스트나 흐름을 만들어낸다.

* 정석파 (`map`): Java(Stream), JavaScript, Python 등
* 데이터베이스파 (`Select`): C# (LINQ)
* 공학파 (`transform`): C++ (STL)

<br>

**핵심 엔진: 지연 평가 (Lazy Evaluation)**
* **Eager (성격 급함):** 호출 즉시 모든 데이터를 계산한다. 메모리 사용량이 높다.
* **Lazy (느긋함):** "다음 값 줘(`next`)"라고 요청할 때까지 계산을 미룬다. 메모리를 획기적으로 절약할 수 있다. (Python 3 기본, Java Stream, C# LINQ, C++ Views 등)

<br>

**파이썬의 구현: 이터레이터와 제너레이터**  
파이썬에서 Map 객체의 실체는 리스트(샌드위치)가 아니라 이터레이터(교환권)이다.
책(List)이 아니라 북마크(Iterator)이기 때문에 `m[0]`처럼 인덱싱을 시도하면 `Not Subscriptable` 에러가 발생한다. 오직 전진(`Next`)만 가능하며, 한 번 순회하면 소멸하는 일회용이다.

※ 이터레이터(Iterator) vs 제너레이터(Generator)
* **이터레이터:** `__next__` 메서드를 가진 모든 객체를 뜻하는 개념이자 인터페이스이다. 직접 구현하기 복잡하다.
* **제너레이터:** `yield`를 사용해 만든 함수로, 이터레이터를 아주 쉽게 구현할 수 있게 해주는 도구이자 부분집합이다.

<br>

**동작 원리: 상태 보존 (State Preservation)**  
지연 평가가 가능한 근본적인 이유는 실행 상태를 저장하기 때문이다.  
일반 함수는 `return`으로 값을 반환하면 스택 프레임을 폭파(Reset)시켜 재호출 시 백지상태가 된다.  
반면 제너레이터는 `yield`로 값을 건네주면 스택 프레임(변수 값과 현재 코드 위치)을 얼려서 힙(Heap)에 보관(Save)한다.   이후 `next()`가 호출되면 이를 녹여서 이어서 실행(Resume)하는 원리로 동작한다.
