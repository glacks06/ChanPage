## 스레드(Thread)의 개념과 멀티스레딩

프로그램(프로세스)을 '주방'이라고 가정할 때, 스레드는 주방에서 일하는 '요리사'에 비유할 수 있다.

* **싱글 스레드 (Single Thread):** 1명의 요리사(CPU)가 재료 손질부터 요리, 설거지까지 순차적으로 혼자 처리하는 방식이다. 충돌은 없지만 작업 속도가 느리다.
* **멀티 스레드 (Multi Thread):** 여러 명의 요리사가 역할을 분담하여 동시에 일을 진행하는 방식이다. 작업 속도가 매우 빠르다.

<br>

## 문제 발생: 경쟁 상태 (Race Condition)

요리사가 여러 명일 때, 도마(공유 데이터)가 하나뿐이라면 심각한 문제가 발생한다.
예를 들어 도마 위에 양파가 1개 있을 때, 요리사 A와 B가 동시에 양파를 집으려고 하면 재고가 마이너스가 되거나 데이터가 중복 처리되는 등 기이한 버그가 발생한다.
이처럼 여러 스레드가 동시에 공유 자원에 접근하여 결과를 예측할 수 없게 되는 상황을 **경쟁 상태(Race Condition)**라고 한다.

<br>

## 해결책: 락(Lock) 메커니즘

이러한 데이터 난장판을 막기 위해 Lock(자물쇠) 개념이 등장한다. 공유 데이터 앞에 문을 달고 자물쇠를 채우는 원리이다.

* 자물쇠를 열 수 있는 열쇠(Key Object)는 단 하나뿐이다.
* 스레드 A가 열쇠를 쥐고 문을 잠근 뒤, 내부에서 안전하게 데이터를 처리한다.
* 그동안 스레드 B는 문 밖에서 스레드 A가 작업을 마치고 열쇠를 제자리에 둘 때까지 대기 상태로 머문다.

## C#에서의 Lock 구현과 내부 동작 원리 (심화)

C#에서는 `lock`이라는 키워드를 통해 동기화를 매우 쉽게 구현할 수 있다. 보통 아무런 기능이 없는 빈 객체(`private readonly object key = new object();`)를 열쇠로 사용한다.

`lock (key)` 코드를 만나면 C# 컴파일러는 `key` 객체의 헤더(Sync Block)를 확인한다.
* **명패 걸기:** 헤더가 비어있으면 현재 스레드 ID를 기록하여 잠금에 성공한다. 만약 다른 스레드 ID가 적혀있다면 운영체제에 대기를 요청하고(Sleep) 대기실로 이동한다.

※ Monitor 클래스와 안전장치:
사실 `lock` 키워드는 개발자를 위한 단축어(Syntactic Sugar)일 뿐이며, 실제로는 `Monitor` 클래스를 사용한 코드로 변환되어 동작한다.

```csharp
Monitor.Enter(key); // (1) 문 잠그기 시도 (못 잠그면 대기)
try {
    // 안전 구역 (Critical Section)
    // 요리 중...
}
finally {
    Monitor.Exit(key);  // (2) 문 열기 (무조건 실행)
}
```

여기서 `finally` 구문의 존재가 매우 중요하다. 내부 구역에서 예외(에러)가 발생하여 작업이 비정상적으로 중단되더라도, 쥐고 있던 열쇠는 반드시 반납하고 종료되도록 보장하기 때문이다. 그래야만 밖에서 대기하던 다른 스레드가 정상적으로 진입할 수 있으며, `lock` 키워드는 이 안전장치를 자동으로 구성해 준다.
