Git에 대해 공부하던 중 블롭(Blob)이라는 단어를 발견해서 공부하게 되었다.

결과적으로 블롭은 Git이 commit을 저장하는 과정에서 디렉터리들을 어떠한 방식으로 구성하고 저장하는지와 연관이 되어있는 개념이다.

이번 기회에 알아보고자 한다.

## 블롭(Blob)

Blob은 Binary Large Object의 약자로 해석하면 '큰 이진 데이터 덩어리'이다.

git에서 블롭은 파일의 실제 내용을 저장한다.

이때 블롭은 이름의 약자에서도 알 수 있듯이 객체이다.

블롭의 특징은 파일 이름, 경로, 권한 등의 메타 정보는 저장하지 않는다는 것이다.

또한 각 블롭은 SHA-1 해시값을 가져서 각각을 식별할 수 있는 식별자로 사용한다.

다시 정리하면 블롭 객체는 파일의 내용만을 저장하는 객체이고, 각 객체마다 SHA-1 해시값을 식별자로 사용한다.

여기서 블롭 객체는 중복된 객체가 생성되지 않는다.

SHA-1 해시값을 생성할 때 파일 크기와 내용을 시드값으로 계산한다.

즉, 같은 내용의 파일에 대해서는 해시값이 동일하게 나온다.

그렇다면 블롭 객체도 정확히 같은 내용의 파일이라면 해시값도 같게 나온다.

이러면 하나의 블롭 객체만 있어도 되지 않은가?

굳이 똑같은 블롭 객체가 여러 개 존재할 필요는 없지 않은가?

그래서 블롭 객체가 생성될 때 같은 해시값을 가지는 블롭 객체가 있는지 확인하고, 같은 해시값을 가지는 블롭 객체가 있다면 같은 블롭 객체를 생성하지 않고 이미 있는 블롭 객체를 참조한다.

같은 해시값을 가지는 블롭 객체가 없다면 정상적으로 블롭 객체를 생성한다,

불필요한 메모리를 사용하지 않는 매우 효율적인 방식이라는 생각이 든다.

_※ 생성된 블롭 객체는 .git/objects/ 디렉터리에 저장된다._

블롭 객체가 생성되는 시점은 git add 명령어를 사용했을 때이다.

git add 명령어로 인해 파일이 stage에 올라가면 블롭 객체가 생성된다.  
<br>

## 트리(Tree)

파일을 저장하는 것은 블롭이였다.

즉, git의 내부 데이터 구조에서 블롭은 파일과 비슷한 것이다.

트리는 디렉터리와 비슷하다.

트리는 해당 트리와 연결되는 각 블롭과 하위 트리를 참조한다.

![트리 그림](/images/tech/git/git-blob-tree1.png)

위 그림이 이해가 잘 안 된다면 아래의 상황과 비슷하다고 보면 된다.

![트리 그림](/images/tech/git/git-blob-tree2.png)

트리가 생성되는 시점은 git commit 명령어를 사용하였을 때이다.

git commit 명령어를 사용하면 staging 영역의 상태를 바탕으로 트리 객체가 생성된다.

이 트리 객체는 커밋 객체에 포함되어 해당 시점의 프로젝트 구조를 기록한다.

트리 객체도 블롭 객체와 마찬가지로 동일한 내용(디렉터리 구조, 파일명, 권한, 참조된 객체)을 가질 경우 중복되어서 생성되지 않고 커밋이 같은 트리 객체를 참조하게 된다.

디렉터리 구조가 같다면 트리 객체의 해시값도 같기 때문이다.  
<br>

## 커밋(commit)

사용자가 git commit 명령어를 사용하면 커밋 객체를 생성한다.

이 커밋 객체는 트리 객체를 참조함으로써 프로젝트의 해당 시점인 스냅샷과 메타 정보를 저장한다.

그림으로 표현하면 다음과 같다.

![트리 그림](/images/tech/git/git-commit.png)

물론 그림상에서는 블롭 객체와 트리 객체가 재사용되는 부분은 표현되지 않았다.

결국 git의 내부 데이터 구조도 우리가 흔히 보는 파일 탐색기의 구조와 같다는 사실을 알 수 있다.  
<br>

## 태그(tag)

태그는 특정 커밋에 이름을 붙여서 해당 시점을 쉽게 찾고 관리할 수 있도록 하는 기능이다.

v1.0, v2.0과 같이 소프트웨어의 릴리스 버전을 표현할 때 많이 사용한다.

태그는 브랜치와 달리 한번 붙여지면 위치가 바뀌지 않아서 새로운 커밋이 추가되어도 이동하지 않는다.

커밋에 태그를 붙이면 해당 커밋을 태그 이름으로 쉽게 체크아웃하거나 참조할 수 있다.

그래서 변경 사항에 대한 내용을 적거나 보다 더 구체적으로 커밋을 설명해야 하는 커밋 메시지와는 차이를 보인다.

태그의 종류로는 Lightweight 태그와 Annotated 태그 두 가지이다.

Lightweight 태그는 태그 이름만 추가하는 것이며 아래와 같은 명령어로 추가한다.
```
git tag <태그이름>
```
Annotated 태그는 태그 이름뿐만 아니라 작성자, 날짜, 메시지 등의 추가 정보가 저장된다.

아래와 같은 명령어로 추가한다.
```
git tag -a <태그이름> -m "태그 메시지"
```   
<br>

```
git checkout <태그이름>
```
위와 같은 명령어를 통해 체크아웃이 가능하다.

이때 주의해야 할 점이 있다.

태그 이름으로 checkout 하여 HEAD를 이동하면 항상 detached HEAD 상태가 된다.

설령 그 자리에 브랜치가 자리 잡고 있더라도 detached HEAD 상태가 되는 것이다.

HEAD가 브랜치를 가리키고 있는 상태는 HEAD가 브랜치를 통해 간접적으로 커밋에 접근할 수 있는 상태이다.

'간접적'으로 접근하는 상황과 다르게 태그를 통해 checkout을 하여 HEAD를 옮기면 HEAD는 그 위치의 커밋을 '직접적'으로 가리키게 된다.

즉, 브랜치를 통해서 접근하는 것이 아니기 때문에 HEAD가 detached 상태가 되는 것이다.

그래서 태그 이름으로 checkout을 한 후에 status를 확인하면 아래와 같은 메시지가 출력된다.

![트리 그림](/images/tech/git/git-tag.png)

그럼  다시 attached 상태가 되게 하려면 어떻게 해야 할까?

두 가지의 방법이 있다.

하나는 브랜치로 다시 checkout 하는 방법이고, 하나는 현재 위치에 브랜치를 새로 생성하는 것이다.

현재 위치에 브랜치를 새로 생성하면 자동으로 HEAD가 생성된 현재 브랜치를 가리키게 되어서 attached 상태가 된다.  
<br>

## 마무리

깃의 내부 데이터 구조를 구성하는 요소는 블롭, 트리, 커밋, 태그가 있었다.

블롭 객체, 트리 객체, 커밋 객체들이 디렉터리 구조와 상태를 저장하는 방식을 알아보았고, 태그를 이용할 때의 주의사항도 알아보았다.

이 글을 작성하면서 HEAD의 detached, attached 상태에 대해 알게 되었고 더 공부해 볼 여지가 있음을 깨닫게 되어서 관련한 자료를 더 찾아볼 예정이다.
