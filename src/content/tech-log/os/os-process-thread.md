32비트 컴퓨터는 최대 32비트 만큼의 주소를 표현할 수 있다.   
이때 컴퓨터는 32비트의 주소를 처리하는 동시에 명령까지 내려햐하는데, 이 모든걸 32비트안으로 표현시키는 것인가에 대한 궁금증에서 생긴 조사이다.

<br>

## 프로세스와 스레드의 기본 개념과 관계

| 구분 | 프로세스(Process) | 스레드(Thread) |
|---|---|---|
| 정의 | 보조기억장치에 있는 프로그램이 운영체제로부터 메모리와 자원을 할당받아 실행 중인 상태 | 프로세스 내부에서 실행되는 실제 작업의 흐름 |
| 성격 | 운영체제가 자원을 분배하는 가장 작은 단위 | CPU가 작업을 처리하는 최소 실행 단위 |
| 비유 | 독립된 건물과 설비를 갖춘 '공장' | 공장 안에서 일하는 '일꾼' |

프로세스는 최소 1개의 스레드(메인 스레드)를 반드시 포함하며, 하나의 프로세스 안에 여러 개의 스레드가 존재할 수 있다(멀티스레딩).

<br>

## 메모리 구조와 자원 공유

프로세스와 스레드의 가장 핵심적인 기술적 차이는 "메모리를 어떻게 나누어 쓰는가"에 있다.

운영체제는 각 프로세스에게 완벽히 분리된 가상 메모리 공간을 할당한다.
* **Code:** 실행할 프로그램의 코드
* **Data:** 전역 변수, 정적 변수
* **Heap:** 동적 할당 영역 (런타임 시 크기 결정)
* **Stack:** 지역 변수, 함수 호출 정보, 리턴 주소

반면, 같은 프로세스 내의 스레드들은 효율성을 위해 메모리의 일부만 공유하고 일부는 독립적으로 가진다.

| 영역 구분 | 포함되는 메모리 영역 | 특징 |
|---|---|---|
| 공유 영역 | Code, Data, Heap | 공장의 설비와 자재를 공유하듯 모든 스레드가 함께 사용하며 데이터를 쉽게 주고받음 |
| 독립 영역 | Stack, PC 레지스터 | 각 스레드는 자신만의 스택과 현재 실행 위치(PC)를 가짐 |

<br>

## 스레드가 독립적인 스택(Stack)을 가지는 이유

스택은 함수 호출 시 돌아가야 할 주소(Return Address)와 지역 변수를 저장하는 임시 작업 공간이다.
만약 스레드들이 스택을 공유한다면 문맥(Context)이 섞여 이전 상태를 복구할 수 없게 되며, 리턴 주소가 섞여 엉뚱한 코드로 점프하는 프로그램 충돌이 발생한다.

※ 결론: "독립적인 실행 흐름"을 유지하고, 언제든 멈췄다가 다시 그 지점부터 실행(Context Switching)하기 위해서는 독립적인 스택이 필수적이다.

<br>

## 실행 제어: 동시성과 병렬성

여러 스레드가 동시에 실행된다는 것은 하드웨어 환경에 따라 다른 의미를 가진다.


| 구분 | 환경 | 작동 방식 | 특징 |
|---|---|---|---|
| 동시성 (Concurrency) | 싱글 코어 | 시간을 아주 잘게 쪼개어(Time Slicing) 스레드를 번갈아 실행 | 물리적으로는 하나씩 실행되나 전환 속도가 빨라 동시에 실행되는 것처럼 보임 |
| 병렬성 (Parallelism) | 멀티 코어 | 각 코어가 스레드를 하나씩 맡아 물리적으로 동시에 실행 | 실제 물리적인 동시 처리 |

※ 멀티스레딩은 하드웨어 코어 수와 관계없이 논리적으로 "여러 작업이 동시에 진행 상태에 있다"는 것을 의미한다.

<br>

## 실전 사례: 웹 브라우저의 작동 원리

크롬(Chrome) 브라우저는 안정성을 위해 탭(Tab)마다 별도의 프로세스를 생성하는 멀티 프로세스 모델을 사용한다. 탭이 스레드로 구현되었다면 하나의 탭이 오류로 멈출 때 브라우저 전체가 종료되기 때문이다.

유튜브 영상을 보며 대용량 파일을 다운로드하는 상황을 통해 프로세스와 스레드의 실제 작동을 단계별로 확인할 수 있다.

<br>

**1단계: 프로그램 실행 (프로세스 생성)**
사용자가 브라우저를 실행하면 운영체제는 하드디스크의 실행 코드를 읽어들인다.
이때 운영체제는 프로세스를 관리하기 위해 프로세스 ID 등의 정보가 담긴 프로세스 제어 블록(PCB)을 생성한다.

<br>

**2단계: 프로세스 자원 할당**
프로그램이 프로세스가 되면 운영체제는 RAM에 독립적인 가상 메모리 공간(Code, Data, Heap, Stack)을 할당한다. OS는 프로세스에게 큰 덩어리의 자원을 부여한다.

<br>

**3단계: 스레드의 생성과 동시 작업 (멀티스레딩)**
기본적으로 UI를 그리는 메인 스레드가 생성되며, 복잡한 동시 작업을 위해 추가 스레드들이 생성된다. OS의 스케줄러는 이 스레드들에게 CPU 사용 시간을 잘게 쪼개어 분배한다.

* **스레드 A (메인 UI 스레드):** 사용자의 마우스 클릭, 스크롤 이벤트를 처리하고 화면을 렌더링한다. 자신만의 Stack을 가져 UI 처리 함수를 독립적으로 호출한다.
* **스레드 B (네트워크 스레드):** 백그라운드에서 대용량 파일을 다운로드한다. 스레드 B가 작업하는 동안에도 스레드 A는 멈추지 않아 다운로드 중에도 브라우저 스크롤이 부드럽게 유지된다.
* **스레드 C (영상 디코딩 스레드):** 영상 데이터를 화면에 뿌릴 수 있는 이미지로 변환하는 복잡한 CPU 연산을 수행한다.

※ 자원 공유와 독립성의 조화: 스레드 B가 다운로드한 데이터는 공유 영역인 Heap에 저장되므로, 스레드 A가 이를 즉시 확인하고 "다운로드 완료" 알림창을 띄울 수 있다. 동시에 스레드 C가 복잡한 연산으로 자신의 Stack을 깊게 사용하더라도 스레드 A의 Stack에는 전혀 영향을 주지 않아 안정적인 독립 실행이 보장된다.
