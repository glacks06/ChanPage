## 1. ⚙️ 하드웨어적 배경: CPU의 메모리 접근 방식

CPU는 메모리를 1바이트씩 개별적으로 읽는 것이 아니라, 워드(Word) 단위로 덩어리째 읽어 들인다. (32bit CPU는 4바이트, 64bit CPU는 8바이트)
데이터 버스(Data Bus)가 물리적으로 해당 비트 수만큼 연결되어 있기 때문에, 주소 또한 4의 배수(0, 4, 8...)로 접근하는 것이 기본 원칙이다.

<br>

## 2. ⚡ 정렬된 주소와 정렬되지 않은 주소의 속도 차이

CPU가 메모리에 접근할 때 주소의 정렬 여부에 따라 처리 속도에 큰 차이가 발생한다.


| 구분 | 정렬된 주소 (Aligned/Even) | 정렬되지 않은 주소 (Unaligned/Odd) |
|---|---|---|
| **상황** | 4바이트 정수가 `0x0000`번지 등 딱 맞아떨어지는 곳에 저장된 경우 | 4바이트 정수가 `0x0001`번지에 저장되어 두 덩어리(`0x0000`, `0x0004`)에 걸쳐 있는 경우 |
| **동작** | 한 번의 메모리 읽기 명령(Fetch)으로 데이터 덩어리를 온전히 가져옴 | 두 번의 메모리 읽기 후, CPU 내부에서 비트 이동(Shift) 및 병합(Merge) 연산 수행 |
| **결과** | 단 1번의 사이클로 처리 완료 (가장 빠름) | 메모리를 두 번 읽고 추가 연산까지 필요하므로 최소 2배 이상 느림 |

<br>

## 3. 🛑 하드웨어 예외와 에러 발생 원인 (Bus Error)

정렬되지 않은 메모리 접근 시, CPU 아키텍처에 따라 대응 방식이 다르다.

* **x86 계열 (Intel/AMD):** 하드웨어가 복잡하더라도 사용자 편의를 위해 느리지만 두 번 읽기를 통해 어떻게든 처리를 지원한다.
* **RISC 계열 (ARM 등):** 하드웨어 단순화와 고성능을 최우선으로 추구하므로, 정렬되지 않은 주소 접근 시 즉시 하드웨어 예외(Bus Error, SIGBUS)를 발생시키고 프로그램을 강제 종료한다.

<br>

## 4. 🧩 컴파일러의 대응: 패딩(Padding)

이러한 속도 저하와 하드웨어 에러를 원천적으로 방지하기 위해, 컴파일러는 변수들 사이에 '패딩(Padding)'이라는 빈 공간을 집어넣어 강제로 메모리 주소를 맞춘다.

※ 패딩의 목적: 메모리 공간을 일부 낭비하더라도, 데이터 접근 속도를 극대화하고 시스템의 전반적인 안정성을 확보하는 것이 훨씬 이득이기 때문이다.
