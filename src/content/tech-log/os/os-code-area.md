## 정의 및 역할

Text Segment(Code 영역)는 C언어 소스 코드가 컴파일러에 의해 번역된 기계어 명령어(Machine Code Instruction)들이 저장되는 공간이다.
CPU는 이 영역에서 명령어를 하나씩 읽어와(Fetch) 연산을 수행하므로, 프로그램의 전체적인 '작업 지시서' 역할을 담당한다.

<br>

## 핵심 특징: 불변과 공유

이 영역의 가장 큰 특징은 프로그램의 안정성과 메모리 효율을 위한 '불변'과 '공유'이다.

* **읽기 전용 (Read-Only):** 프로그램 실행 도중 코드가 변조되는 것을 막기 위해 쓰기 권한이 없다. 강제로 쓰기를 시도하면 Segmentation Fault가 발생한다.
* **공유 가능 (Sharable):** 같은 프로그램을 여러 개 실행해도, 메모리 효율을 위해 하나의 Text Segment를 모든 프로세스가 공유하여 사용한다.
* **고정 크기:** 메모리 크기는 컴파일 타임에 결정되며 런타임에는 변하지 않는다.

<br>

## 실행 흐름의 제어 (PC 레지스터)

명령어의 실행 흐름을 제어하는 것은 CPU 내부의 PC(Program Counter) 레지스터이다.
다음에 실행할 명령어의 주소를 가리키며, 기본적으로는 순차 증가하지만 `if`문이나 함수 호출(`Call`) 등을 만나면 특정 주소로 점프(Jump)하여 프로그램의 흐름을 바꾼다.

※ Text Overflow: 런타임에 터지는 스택 오버플로우와 달리, 주로 컴파일/링크 타임(Build-time)에 발생하여 실행 파일 생성 자체를 실패하게 만든다. 임베디드 시스템의 저장 공간 한계나, 함수 호출 거리가 명령어의 표현 범위(±2GB)를 벗어날 때 링커 에러로 나타난다.

<br>

## 명령어 주소 지정 방식과 실행 메커니즘

x86-64 등 현대 아키텍처는 함수 호출(CALL) 시 전체 주소(Absolute) 대신, '현재 위치(PC)로부터의 거리(Offset)'를 저장하는 상대 주소 지정 방식(Relative Addressing)을 표준으로 사용한다.
이때 오프셋 저장 공간으로 32비트(4바이트)를 사용하므로, 32비트 부호 있는 정수의 최대 범위인 ±2GB 이내로만 점프가 가능하다.

이러한 상대 주소는 명령어 길이를 줄이기 위한 '압축된 표현'일 뿐이다. CPU가 실제로 메모리에 접근하기 위해서는 정확한 물리적 주소(Effective Address)가 필요하다.
따라서 명령어가 실행되는 순간 산술논리연산장치(ALU)가 필수적으로 개입하여 `[현재 PC 값] + [명령어 내의 오프셋]`을 더하는 연산을 수행한다.
당연하게도 이 덧셈 연산 과정에서 ALU의 비트 수(32bit/64bit)에 따른 물리적 주소 공간의 한계가 적용된다. (즉, 오프셋을 써도 32비트 ALU로는 4GB 너머의 주소를 계산해낼 수 없다.)

<br>

## 설계 철학: 왜 32bit 상대 주소를 쓰는가?

전체 절대 주소를 쓰지 않고 굳이 32bit 상대 주소를 사용하는 이유는 다음과 같다.

| 설계 철학 | 설명 |
|---|---|
| 지역성의 원리 (Locality) | 통계적으로 프로그램의 점프(분기, 함수 호출) 99% 이상은 가까운 거리 내에서 발생하므로, 굳이 64비트 전체 주소를 쓸 필요가 없다. |
| 코드 밀도 (Code Density) | 명령어 길이를 대폭 줄여(10byte → 5byte) 명령어 캐시(Instruction Cache) 효율을 극대화하고 실행 속도를 높인다. |
| 위치 독립 코드 (PIC) | 코드가 메모리의 어느 위치에 로딩되더라도 "상대적 거리"는 변하지 않으므로 코드를 수정(재배치)할 필요가 없다. |
