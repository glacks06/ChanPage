## 전체 프로세스 메모리 구조에서의 위치

프로그램이 실행될 때 운영체제가 할당하는 메모리는 크게 구획이 나뉜다.  
기계어 코드가 들어가는 Code 영역, 동적 할당을 위한 거대한 창고인 Heap 영역, 함수 호출과 흐름을 제어하는 Stack 영역이 있다.  
그리고 전역 변수와 정적(Static) 변수가 저장되는 공간이 바로 Data 영역과 BSS 영역이다.

<br>

## 두 영역의 결정적 차이: 초기화 여부

이 두 영역을 나누는 유일한 기준은 오직 변수의 초기화 여부이다.

| 영역 | 저장 대상 | 예시 |
|---|---|---|
| Data Segment | 코드에서 명시적으로 초기값(10, 100 등)을 지정한 전역 변수나 정적 변수 | `int g_val = 100;` |
| BSS Segment | 초기값을 지정하지 않은 전역 변수나 정적 변수 (프로그램 실행 시 자동으로 0 초기화) | `int g_no_val;` |

<br>

## 실행 파일 용량에 미치는 영향 (분리 목적)

왜 굳이 두 영역을 나누었을까? 그 해답은 실행 파일(.exe)의 용량 최적화에 있다.
Data Segment는 변수가 가진 초기값 자체가 중요하므로 그 값을 실행 파일에 그대로 기록해야 한다. 따라서 변수가 많아질수록 파일 용량도 정직하게 커진다.

하지만 BSS Segment의 변수들은 어차피 실행될 때 0으로 시작할 것들이다.
따라서 파일 내부에 0을 수만 개 기록하는 대신, "BSS 영역의 크기는 O바이트다"라는 요약 정보만 파일 헤더에 기록한다. 이를 통해 하드디스크 공간을 획기적으로 절약할 수 있다.

<br>

## BSS의 0 초기화 과정 (런타임)

BSS 영역의 변수들이 0으로 초기화되는 과정은 운영체제가 변수를 하나하나 검사하며 이루어지는 것이 아니다.

컴파일 타임에 컴파일러와 링커가 초기화되지 않은 변수들을 BSS 영역으로 미리 분류하고, 실행 파일 헤더에 필요한 전체 크기만 기록해 둔다.
이후 런타임에 프로그램을 실행하면, 운영체제의 로더(Loader)가 이 헤더를 읽어 해당 크기만큼 메모리를 크게 할당한 뒤, 그 공간을 통째로 0으로 밀어버린다(Zero-fill).

※ 로더(Loader): 하드디스크의 프로그램과 데이터를 읽어 메모리에 적재하고 실행을 준비하는 운영체제의 핵심 구성 요소이다.

<br>

## 메모리 배치와 정렬 (Alignment)

BSS 영역 내에 할당된 변수들은 인접하게 배치되지만, CPU의 처리 속도 향상을 위해 빈틈없이 빽빽하게 붙어있지는 않는다.

여기에도 CPU가 한 번에 읽기 좋은 단위(4바이트 등)로 주소를 맞추기 위해 변수 사이에 의미 없는 빈 공간을 집어넣는 패딩(Padding)이 적용된다.

※ 재배치(Reordering): 컴파일러는 이러한 패딩으로 인한 메모리 낭비를 줄이기 위해, 프로그래머의 원래 선언 순서와 상관없이 변수의 크기별로 순서를 바꿔서 재배치하기도 한다.
