도커에 대해 알아보면서 여러가지 질문사항이 생겼다.
인공지능과 계속적으로 대화하면서 질문들에 대한 답을 얻었고 그 내용을 아래에 정리했다.

<br>

## 1. 🍎 macOS와 '숨겨진 VM'의 관계
- Q: macOS도 리눅스 VM이 필요한가? 유닉스(Unix) 계열이라 괜찮지 않나?
- A: macOS도 윈도우처럼 경량화된 리눅스 VM이 필요하다. 이유는 macOS와 리눅스의 '커널(Kernel)'이 다르기 때문이다. macOS는 'XNU' 커널을 사용하고, 리눅스는 '리눅스(Linux) 커널'을 사용한다. `ls`, `cd` 같은 명령어(유닉스 명령어)가 비슷하더라도, 커널이 사용하는 '기초 언어(시스템 콜)'가 다르다. 따라서 'Docker Desktop for Mac'도 macOS의 가상화 기술(Hypervisor.framework)을 이용해 '숨겨진 리눅스 VM'을 띄우고, 그 위에서 리눅스 컨테이너를 실행한다.

<br>

## 2. 📜 도커와 OS 라이선스 문제
- Q: 유료 OS(윈도우 서버 등)도 도커에서는 무료로 쓸 수 있는가?
- A: 라이선스 비용은 면제되지 않는다. 도커라는 '도구'와 그 안에서 실행되는 '소프트웨어'의 라이선스는 별개이다.
<br>

- Q: 라이선스는 어떻게 처리되는가?
- A: SaaS 구독형(예: Red Hat - RHEL)은 유효한 유료 구독 계정으로 `docker login`을 해야만 공식 이미지를 pull(다운로드)할 수 있다. 호스트 라이선스 기반(예: 윈도우 컨테이너)은 라이선스가 컨테이너가 아닌 '호스트(Host) OS'를 따른다. 즉, 내 PC(호스트)가 정품 윈도우 Pro 또는 Server 라이선스를 가지고 있어야만 윈도우 컨테이너를 실행할 권한이 인정된다.
<br>

- Q: 그럼 이미지를 파일로 받으면 무료로 쓸 수 있는가?
- A: 기술적으로 `docker load`를 통해 실행할 수는 있어도, 이는 라이선스 위반(불법 복제)이다.
<br>

- Q: 도커가 라이선스를 검사하는가?
- A: 도커 자체가 '라이선스 경찰'은 아니다. 라이선스 집행은 OS 공급사의 정책에 따른다. 윈도우는 호스트 OS의 라이선스를 확인하고, Red Hat은 다운로드 시 인증하거나 실행 후 업데이트 및 기술 지원 인증에 실패하게 만든다. 도커는 COPY 명령어를 충실히 따를 뿐, 개발자가 불법 파일을 포함시킨 책임은 전적으로 개발자에게 있다.
<br>

## 3. 🏗️ 도커 이미지의 생성과 공유
- Q: 이미지를 만들 때, 텍스트 파일만 쓰는가, 명령어를 실행하는가?
- A: 둘 다이다. 개발자는 Dockerfile이라는 '텍스트 파일(설계도)'을 작성한다. `docker build` 명령을 실행하면, 도커가 그 텍스트 파일 안의 명령어(예: `RUN npm install`)를 순서대로 '실제로 실행'하여 이미지를 만든다.
<br>

- Q: 이미지를 다른 사람에게 넘길 수 있는가?
- A: 넘길 수 있다. 하지만 받는 사람의 PC에도 '도커 엔진'이 설치되어 있어야 한다. 방법 1(파일)은 `docker save`로 `.tar` 파일을 만들고, `docker load`로 불러오는 것이다. 방법 2(권장)는 '도커 허브(Docker Hub)' 같은 공용 이미지 레지스트리에 `docker push`로 올리고, 다른 사람이 `docker pull`로 다운로드하는 방식이다.
<br>

## 4. LAYER 도커 이미지의 '레이어(Layer)' 구조
- Q: 도커 이미지는 다른 이미지들이 결합된 것인가?
- A: 그렇다. 도커 이미지는 '통짜 파일'이 아니라, '레이어(Layers)'라고 불리는 여러 개의 읽기 전용 이미지 조각들이 '겹쳐진(Stacked)' 묶음이다. 비유하자면 '투명 OHP 필름'을 겹쳐놓은 것과 같다. (가장 아래 필름: `FROM ubuntu`), (그 위 필름: `RUN apt install nginx`), (그 위 필름: `COPY . /app`) 형태로 구성된다.
<br>

- Q: 레이어 방식의 이점은 무엇인가?
- A: 빌드 속도(캐싱)와 디스크 효율성(공유)이 뛰어나다. `docker build`를 다시 할 때, 변경되지 않은 레이어는 '캐시(Cache)'에서 그대로 가져와 재사용한다. 또한 `nginx` 이미지와 `node` 이미지가 둘 다 `ubuntu`를 기반으로 한다면, 디스크에 `ubuntu` 베이스 레이어는 단 1개만 저장되고 두 이미지가 이를 공유한다.
<br>

## 5. 📦 깃허브(GitHub) vs 도커 허브(Docker Hub)
- Q: 도커 이미지를 깃허브에 올릴 수는 없는가?
- A: 올릴 수 없고 올려도 소용없다. 두 저장소는 목적이 다르다. 깃허브(GitHub)는 '소스 코드(설계도)'를 저장하고, 도커 허브(Docker Hub)는 빌드된 '도커 엔진 전용 이미지(완성품)'를 저장한다.
<br>

- Q: 깃허브에 git push로 이미지 파일을 올리면 어떻게 되는가?
- A: 실패한다. 깃허브는 단일 파일 100MB 용량 제한이 있다. Git은 텍스트 변경 이력을 추적하는데, 이미지는 거대한 단일 파일이라 이력이 쌓이면 저장소에 과부하가 걸려 비효율적이다. 또한 `docker pull`로 다운로드할 수도 없어 무용지물이다.
<br>

- Q: 깃허브 생태계에서 이미지를 관리할 방법은 없는가?
- A: 깃허브도 'GitHub Container Registry (GHCR, ghcr.io)'라는 별도의 이미지 레지스트리를 제공하므로, `docker push`는 이곳으로 해야 한다.
<br>

## 6. 🔄 소스 코드 수정 시 개발 워크플로우
- Q: 이미지를 만든 후 소스 코드를 수정하면 어떻게 하는가?
- A: 목적에 따라 두 가지 상황으로 나뉜다. 개발 완료 후 배포나 공유가 목적이라면 `docker build`를 다시 실행하여 새 이미지를 만들어야 한다. 이때 레이어 캐싱 덕분에 변경된 코드 부분만 빠르게 다시 빌드되며, 기존 컨테이너를 중지하고 새 이미지로 컨테이너를 실행한다. 실시간 개발 중이라면 매번 빌드하는 것이 비효율적이므로 `docker run` 시 '볼륨 마운트 (`-v`)' 옵션을 사용한다.
<br>

- Q: 볼륨 마운트란 무엇인가?
- A: 내 PC(호스트)의 소스 코드 폴더와 컨테이너 내부 폴더를 '실시간 동기화'하는 기능이다. 개발자가 PC에서 코드를 저장하면 즉시 컨테이너 내부 코드에 반영되며, 컨테이너 안의 `nodemon` 등이 변경을 감지하고 서버를 자동 재시작하므로 별도의 빌드가 필요 없다.
<br>

## 7. 🏭 CI/CD 파이프라인과 서비스 분리
- Q: 버전 컨트롤은 깃허브, 푸시는 도커로 하는가?
- A: 그렇다. `git push`(코드를 깃허브에 보냄) 후 `docker push`(이미지를 레지스트리에 보냄)로 이어지는 흐름이 표준이다.
<br>

- Q: CI/CD 파이프라인이란 무엇인가?
- A: 개발 및 배포 과정을 '자동화'한 시스템이다. CI(지속적인 통합)는 `git push` 시 자동화 서버(예: GitHub Actions)가 코드를 가져가 테스트하고 `docker build`까지 실행한다. CD(지속적인 배포)는 CI 성공 시 자동으로 `docker push`를 실행해 이미지를 올리고, 운영 서버에서 `docker pull` 및 `run`까지 자동으로 수행한다.
<br>

- Q: 프론트엔드와 백엔드는 각각 컨테이너를 만드는가?
- A: 반드시 분리해야 한다. 구동 환경이 다르고 독립적인 확장 및 배포가 필요하기 때문이다. `docker-compose.yml`이라는 통합 설계도 파일을 사용하여 다수의 서비스를 한꺼번에 정의하고 `docker-compose up` 명령어로 동시에 관리하는 방식을 사용한다.
<br>

## 8. 🔌 포트(Port) 번호와 연결 방식
- Q: 포트 번호는 모두 외워야 하는가?
- A: 아니다. 표준 포트와 임의 포트로 구분된다. 80(HTTP), 443(HTTPS), 3306(MySQL) 같은 표준 포트는 국제적인 약속이므로 몇 가지만 알아두면 된다. 3000(React), 8080(Node) 등은 개발자가 자유롭게 선택하는 임의 포트이다.
<br>

- Q: `docker run -p 8080:80`의 상세 과정과 이유는 무엇인가?
- A: '호스트 외부 포트(8080)'와 '컨테이너 내부 포트(80)'를 매핑(연결)하는 명령어이다. 컨테이너 내부 포트(`:80`)는 Nginx 이미지처럼 80번을 사용하도록 정해져 있는 반면, 호스트 포트(`8080:`)는 충돌과 권한 문제를 피하기 위해 임의로 지정한다. 1024 미만 포트는 관리자 권한이 필요하고 호스트 PC의 다른 프로그램과 충돌할 수 있기 때문이다. 요청 흐름은 '사용자(localhost:8080 요청) → 호스트 OS → 도커 엔진(포워딩) → 컨테이너(80 응답)' 순으로 진행된다.
<br>

- Q: pm2도 포트를 도커처럼 할당하는가?
- A: 아니다. pm2는 네트워크 격리가 없으므로 `app.listen(8080)`을 실행하면 앱 프로세스가 호스트 PC의 8080 포트를 직접 점유하며 매핑 과정이 없다.
<br>

- Q: 코드를 짤 때 포트를 정하고 실행 시 매핑하는 구조인가?
- A: 그렇다. 코드 작성 시 `app.listen(8080)`처럼 '컨테이너 내부'에서 쓸 포트를 정하고, 실행 시 `docker run -p 3000:8080`처럼 외부 호스트 포트와 내부 컨테이너 포트를 연결한다. 이 매핑이 없으면 외부에서 컨테이너로 접근할 수 없다.
