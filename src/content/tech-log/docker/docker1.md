## 도커의 기본 개념과 핵심 요소

도커는 애플리케이션을 '컨테이너(Container)'라는 단위로 패키징하여 배포하고 실행하는 플랫폼이다.
핵심은 "한 번 만들면 어디서든 실행된다"는 것이다.
개발 환경과 운영 환경의 설정 차이로 발생하는 문제를 해결하기 위해, 실행에 필요한 모든 것을 컨테이너에 담아 밀봉한다.

<br>

※ Dockerfile: 컨테이너를 만드는 설계도이다.

※ Docker Image: 설계도를 바탕으로 만들어진 실행 가능한 템플릿이다. 원본 환경을 1:1로 복제한 스냅샷을 의미한다.

※ Docker Container: 도커 이미지를 실제로 실행한 상태(인스턴스)이다. 하나의 이미지로 여러 개의 컨테이너를 만들 수 있다.

<br>

## 가상 머신(VM)과의 결정적 차이

도커와 가상 머신의 가장 큰 차이는 가상화의 대상과 OS 설치 여부에 있다.

| 구분 | 가상 머신(VM) | 도커(Docker) |
|---|---|---|
| 가상화 대상 | 하드웨어 (Host OS 위에 Guest OS 통째로 설치) | OS 커널 (Host OS의 커널 공유, 파일 시스템만 격리) |
| OS 설치 여부 | 파티션을 나누고 OS 직접 설치 (수 분 소요) | 커널만 제외한 파일 묶음(이미지) 다운로드 및 복사 (수 초 소요) |
| 커널 유무 | 독립된 커널 소유 | 커널 없음 (외부 커널 공유) |
| 실행 결과 | 10개의 앱 실행 시 10개의 무거운 OS 띄움 | 10개의 앱이 1개의 커널을 공유하며 가볍게 실행됨 |

가상 머신은 10개의 앱을 위해 10채의 단독주택을 짓는 것이라면, 도커는 1개의 아파트에 10개의 원룸을 나누는 것과 같다.

<br>

## 윈도우와 macOS에서의 도커 실행 방식

윈도우나 macOS 환경에서 리눅스 컨테이너를 실행할 때, 명령어 처리에 대한 오해가 존재한다.
흔히 윈도우 커널이 리눅스 명령어를 번역해서 실행한다고 생각하지만, 이는 사실이 아니다. 리눅스 명령어는 윈도우 커널이 알아듣지 못한다.

Docker Desktop은 이 문제를 해결하기 위해 보이지 않는 곳에 '진짜 리눅스 커널'을 포함한 경량 VM을 1개 설치하여 실행한다.

<br>

※ 하이퍼바이저(Hypervisor): 이 숨겨진 VM을 만들고 실행하는 소프트웨어 엔진이다.

<br>

실제 실행 흐름은 다음과 같다.
리눅스 명령어는 VM 내부의 리눅스 커널이 100% 직접 처리한다.
이후 파일 저장 등 물리적 하드웨어 접근이 필요해지면 하이퍼바이저가 이 요청을 가로채어 윈도우식 파일 쓰기 요청으로 변환하여 전달한다.
결과적으로 리눅스 명령어는 리눅스 커널이 실행하지만, 실제 자원 제어는 하이퍼바이저를 거쳐 윈도우 커널이 최종 승인하는 방식이다.

<br>

## 실제 사용법과 개발 워크플로우

도커를 사용할 때 `docker run` 명령어에 `-d` 옵션이 없으면 포그라운드 모드로 실행되어 터미널이 먹통처럼 보일 수 있다.  
`-d` 옵션을 주면 백그라운드에서 조용히 실행된다.
 
<br>

개발 시에는 컨테이너 안에서 직접 코딩하는 것이 아니라, 볼륨 마운트(Volume Mount)를 활용한다.
개발자는 로컬 PC에서 코딩하고, 컨테이너는 공유받은 코드를 실시간으로 반영하여 실행한다.
이때 로컬 PC가 아닌 컨테이너(실제 환경)를 기준으로 개발해야 내 PC에서만 작동하는 오류를 개발 첫 단계부터 원천 차단할 수 있다.

<br>

※ 도커 vs pm2: pm2는 이미 갖춰진 OS에서 특정 프로세스만 감시하지만, 도커는 환경 전체(OS 파일, 런타임, 코드 등)를 격리하고 관리한다.

<br>

## 컨테이너의 작동 원리

컨테이너는 OS로부터 '미니 OS'를 할당받는 것이 아니다.
커널의 핵심 기술을 통해 격리된 공간과 자원을 제공받는다.

<br>

※ Namespaces (네임스페이스): 독립된 파일 시스템, 프로세스 ID 등을 통해 컨테이너가 혼자만 존재한다고 착각하게 만드는 격리 공간이다.

※ Cgroups (씨그룹): 해당 컨테이너가 사용할 수 있는 CPU, RAM 등의 자원 할당량을 제한하는 기술이다.
