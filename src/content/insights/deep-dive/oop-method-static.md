## 1. 메서드의 위치 결정 기준 (어디에 넣고 뺄 것인가)
* **정보 전문가 패턴:** 연산에 필요한 데이터를 가장 많이 가진 객체가 그 일을 직접 처리하게 한다.
* **상태 의존성:** 객체의 내부 상태(`this`)를 사용하거나 변경한다면 인스턴스 메서드로 내부에 둔다.
* **단일 책임 원칙 (SRP):** 객체의 본질적인 정체성을 나타내는 로직만 내부에 남기고, DB 저장이나 포맷 변환 같은 인프라적 성격의 로직은 외부(Service, Helper)로 뺀다.
* **값의 조회와 연산:** 내부 변수의 값을 변경하지 않고 단순 연산만 하더라도, 내 데이터를 기반으로 한다면 인스턴스 메서드(혹은 프로퍼티)로 구현하는 것이 정석이다.
* **핵심 철학 ("Tell, Don't Ask"):** 외부에서 데이터를 꺼내서(`get`) 판단하지 말고, 객체에게 직접 하라고 명령(`tell`)한다.

<br>

## 2. 유니티(Unity) 환경에서의 메서드 분리 전략
* **Static Class:** 상태가 없고 어디서든 쓰이는 순수 계산 및 변환 로직을 모아둔다.
* **Extension Method:** 유니티 내장 클래스(`Transform` 등)에 나만의 기능을 덧붙이고 싶을 때 사용한다.
* **Manager / Service:** 게임 전반의 상태를 관리하거나 여러 객체를 조율할 때 사용하며, 주로 싱글톤 패턴을 적용한다.
* **ScriptableObject:** 마스터 데이터(스탯, 설정값)와 그 데이터를 다루는 로직을 함께 묶어 메모리를 최적화한다.
* **일반 C# 클래스 (POCO):** 유니티 엔진의 생명주기(`Update` 등)가 필요 없는 무거운 로직을 분리하여 `MonoBehaviour`를 가볍게 만든다.

<br>

## 3. static 함수와 인스턴스 함수의 구분 기준
* **구분의 절대적 기준:** **"함수 내부에서 객체의 상태(`this`)를 사용하는가?"**이다.
* **static을 쓰는 경우:** 외부 상태에 의존하지 않고 파라미터만으로 결과를 내는 순수 함수, 전역 유틸리티, 팩토리 메서드를 만들 때 사용한다.
* **static을 피해야 하는 경우:** 객체마다 고유한 데이터가 필요할 때, 다형성(오버라이딩)이 필요할 때, 독립적인 유닛 테스트가 필요할 때는 반드시 인스턴스 메서드를 사용한다.

<br>

## 4. static 메서드에서 this를 쓸 수 없는 근본적인 이유
* **컴파일러의 작동 방식:** 원래 인스턴스 메서드는 컴파일러가 몰래 첫 번째 인자로 자기 자신(`this`)의 참조를 전달한다. 하지만 `static` 메서드는 이 '숨겨진 첫 번째 인자 전달' 과정 자체가 생략된 특수한 함수다. 애초에 `this`라는 변수 자체가 전달되지 않으므로 사용할 수 없다.
* **메모리 관점의 한계:** `static`은 실체(인스턴스)가 생성되기 전부터 메모리의 고정 영역에 올라가 있는 '설계도'와 같다. 따라서 호출되는 시점에 수많은 객체 중 어떤 객체를 가리키는지 알 방법이 없다.

<br>

## 5. 타 언어에서의 작동 원리 비교
* **C++ / Java:** C#과 완전히 동일하게 컴파일러가 암시적으로 `this`를 처리한다.
* **Python:** C#의 `this`에 해당하는 `self`를 첫 번째 인자로 **반드시 직접 명시**해야 한다. 이를 통해 "static 메서드는 첫 번째 인자로 자기 자신을 받지 않는다"는 대원칙을 시각적으로 가장 명확하게 보여준다.
* **JavaScript:** `static` 메서드 내부에서 `this` 키워드를 사용할 수는 있으나, 이는 인스턴스가 아닌 **클래스 그 자체**를 가리키므로 다른 언어와 의미가 다르다. 결과적으로 인스턴스의 변수에는 접근할 수 없다는 본질은 같다.
