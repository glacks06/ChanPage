흔히 프로그래밍 언어를 컴파일 언어와 인터프리터 언어로 나눈다.
이렇게 보면 컴파일 언어는 컴파일 과정을 거치고, 인터프리터 언어는 컴파일 과정을 거치지 않는 것처럼 느껴진다.

평소처럼 파이썬으로 코딩을 하다가 문득 떠오른 생각이 있었다.
*'하지만 컴파일은 소스코드를 기계어로 변환하는 작업인데, 파이썬도 기계어로 변환하는 과정이 필요하지 않나?'*
이러한 궁금증을 계기로 조사해 보았다.

결론부터 말하자면 모든 인터프리터 언어가 컴파일 과정을 거치지 않는 건 아니다.
예를 들어 파이썬도 컴파일 과정을 거친다.
다만 컴파일의 결과물, 실행 방식에서 차이가 있다.  
<br>

## 파이썬 실행 과정

우리가 소스코드를 작성하면 Python Compiler가 소스코드를 Byte Code로 변환을 한다.
python.org에서 파이썬을 설치하게 된다면 CPython이라는 compiler가 같이 설치된다.

이렇게 설치된 CPython이 소스코드를 Byte Code로 변환하는 역할을 수행한다. (파일 형식: `.py` -> `.pyc`)
이 대목에서부터 벌써 파이썬이 컴파일 과정을 수행한다는 것을 알 수 있다.

CPython에 의해 소스코드가 Byte Code로 변환되면 해당 코드를 PVM(Python Virtual Machine)이 실행하게 된다.

> 💡 **참고 개념 정리**
> * **Python 인터프리터:** Python Compiler(CPython)와 PVM을 통칭한다.
> * **Byte Code:** 가상 머신이 실행할 수 있는 코드를 말한다. 파이썬 소스 코드가 Byte Code가 되면 파일 형식이 `.pyc`가 된다.
> * **PVM (Python Virtual Machine):** 파이썬을 실행하는 가상 머신이다. 파이썬은 컴파일될 때 CPU가 처리할 수 있는 실행파일이 아닌 가상 머신이 실행할 수 있는 Byte Code 파일이 생성되기 때문에 PVM이 이를 실행한다. 파이썬은 이러한 점 때문에 CPU의 종류와 상관없이 PVM이 있다면 실행이 가능해 플랫폼 독립적이라는 특징을 가진다.

## 그럼 파이썬은 인터프리터 언어일까?

사람들은 프로그래밍 언어의 종류를 컴파일러 언어 vs 인터프리터 언어 구도로 설명하곤 한다.
이러면 마치 인터프리터 언어는 컴파일 과정이 없는 것처럼 느껴진다.

하지만 위에서도 확인했듯이 인터프리터 언어의 대표적인 예시인 파이썬은 컴파일 과정을 거친다.
이를 보고 굉장히 당황스러웠다.

그래서 조사를 해보니 프로그래밍 언어는 컴파일 언어와 인터프리터 언어뿐 아니라 컴파일과 인터프리터 언어가 섞인 형태의 **하이브리드 언어**가 존재한다는 것을 알게 되었다.
파이썬같이 소스코드를 Byte Code로 변환한 후 가상머신이 실행하는 형태가 하이브리드 언어이다.

우리가 생각하는 순수 인터프리터 언어로는 Bash, JavaScript 등이 있다.
따라서 파이썬은 인터프리터 언어라고 하기보다는 하이브리드 언어라고 지칭하는 게 조금 더 맞는 표현인 듯하다.  
<br>

## 프로그래밍 언어를 컴파일 언어, 인터프리터 언어로 구분 짓는 것에 대한 견해

자료를 더 조사해 보던 중 더블린 대학의 Eddie Antonio Santos가 적은 블로그 글을 읽고 매우 흥미로웠다.
프로그래밍 언어를 컴파일 언어, 인터프리터 언어로 나누는 것에 대해 불만이라는 내용이었다.
글쓴이는 파이썬을 토마토에 비유한다.

토마토는 과일일까 채소일까?
과일은 “씨를 가진 자방(子房)이 성숙한 것"이다.
토마토도 씨를 가진 자방이 성숙한 것이다. 그럼 토마토는 과일일까?

미국 농무부는 다른 과일처럼 디저트로 제공되는 것이 아니라 저녁 식사에 주식으로 함께 조리되거나 곁들여져 식탁에 오르기 때문에 채소라는 결론을 내놓았다.

결국 토마토는 과일이기도 채소이기도 하기 때문에 '과채류'라고 불린다.
이 부분에서 토마토는 파이썬과 닮았다.
파이썬도 컴파일 언어라기엔 인터프리터적 특징을 지녔고, 인터프리터 언어라기엔 컴파일 과정을 거친다.

Eddie Antonio Santos는 이러한 부분에서 프로그래밍 언어를 컴파일 언어와 인터프리터 언어로 구분 짓는 것은 잘못되었다고 말한다.  
<br>

> "A language is not inherently compiled or interpreted; whether a language is compiled or interpreted (or both!) is an implementation detail."<br>
> **어떤 언어가 컴파일되거나 인터프리트 되는지는 구현 방식에 따라 달라지는 세부 사항일 뿐이다.**

> "Thinking in such a rigid way limits what we believe a given programming language can do."<br>
> **이처럼 경직된 사고방식은 특정 프로그래밍 언어가 할 수 있는 일에 대한 우리의 인식을 제한한다.**  
<br>


## Eddie Antonio Santos가 생각하는 더 올바른 구분

Eddie Antonio Santos는 [컴파일 vs 인터프리터]가 아닌 언어의 속성(property)을 기준으로 하여 **[정적 속성 vs 동적 속성]**으로 구분해야 하며, 이마저도 이분법적으로 언어가 나뉘는 것이 아닌 스펙트럼이라고 설명한다.

*(※ 여기서 언어의 속성이란 해당 언어의 코드가 어떻게 해석되고 실행되는지 등의 구조적 특징이나 설계 원칙을 말한다.)*

정적 속성과 동적 속성에 대한 설명은 다음과 같다. (From ChatGPT)

| 속성 유형 | 설명 | 예시 언어 | 예시 |
| :--- | :--- | :--- | :--- |
| **정적 속성** | 실행 전에 (즉, 코드만 보고) 분석이 가능한 속성 | C, Rust, Java | 변수 타입 검사, 컴파일 시 에러 감지 |
| **동적 속성** | 실행 중(runtime)에만 결정되는 속성 | Python, Ruby, JavaScript | 동적 타입, 리플렉션, 실행 중 메서드 변경 |

위 기준에 따르면 파이썬은 동적 속성을 많이 지닌 언어인 것이다.

이것이 [컴파일 vs 인터프리터] 구도보다 더 나은 이유는 무엇일까?
그 이유는 컴파일이냐 인터프리터이냐의 차이보다 **얼마나 정적이냐 동적이냐가 실제 프로그래밍에서 더 중요한 차이**이기 때문이다.

정적 속성과 동적 속성에 대한 차이점은 다음과 같다. (From ChatGPT)

| 구분 | 정적 (Static) | 동적 (Dynamic) |
| :--- | :--- | :--- |
| **코드 분석** | 실행 전에 많은 오류를 발견 가능 | 실행 중에만 오류 감지 |
| **타입 시스템** | 변수 타입이 컴파일 타임에 결정됨 | 변수 타입이 런타임에 결정됨 |
| **최적화** | 정적 분석을 통해 최적화 가능 | 최적화가 어려움 |
| **유연성** | 코드가 더 엄격하지만 안정적 | 코드가 더 유연하지만 예측하기 어려움 |

위의 특성들이 실제 프로젝트를 진행할 때 훨씬 더 중요하게 작용하기 때문에 [정적 속성 vs 동적 속성]이라는 기준으로 보는 것이 더 올바르다.
이를 보면 Eddie Antonio Santos가 '어떤 언어가 컴파일되거나 인터프리트되는지는 구현 방식에 따라 달라지는 세부 사항일 뿐이다.'라고 한 것이 이해가 간다.

한마디로 **'만들어지고 구현되는 방식'보다는 '구현된 언어가 어떤 특징을 가지는지'에 초점을 맞춰야 한다는 것**이다.

## 마무리

조사를 하면서 혼란의 연속을 겪은 것 같다.
이제껏 파이썬을 써오면서 단 한 번도 인터프리터 언어에 대해 알아본 적이 없고, 파이썬도 컴파일 과정을 거친다는 사실도 몰랐다는 것이 충격이었다.
또한 [컴파일 언어 vs 인터프리터 언어]와 [언어의 정적 속성 vs 언어의 동적 속성], 이 두 가지 기준의 차이점을 이해하는 과정에서도 다소 헷갈리는 부분이 많았다.

그래도 이번 기회에 프로그래밍 언어에 대해 조금은 더 알아가는 시간을 가질 수 있었고, 앞으로도 내가 당연히 알고 있다고 생각하는 개념에 정말 알고 있는지 질문을 던져보는 시간을 자주 가져야겠다는 생각을 하게 되었다.
