## 무의미한 Getter/Setter를 버리고 Struct를 쓰기
* 데이터가 유효하지 않은 값으로 설정될 가능성이 전혀 없다면(예: 맵 좌표, 이름, 단순 설정값), 그것은 '클래스'가 아니라 단순한 '데이터 구조(Struct)'다.
* 이런 데이터는 굳이 `private`으로 감추고 의미 없는 `getter/setter`를 덧붙여 억지로 클래스인 척 포장할 필요가 없다. 
* 변경 시 논리적 모순을 일으키지 않는 독립적인 데이터(예: 유니티의 `Vector3`, `Color`)는 `public`으로 열어둔 `struct`를 사용하여 직관적이고 가볍게 설계하는 것이 올바른 방향이다.

<br>

## 클래스(Class)의 존재 이유는 '불변성(Invariant)' 유지
* 데이터가 특정 범위나 조건을 벗어나면 시스템에 오류를 일으키는 경우(예: 체력이 0 미만으로 떨어지는 경우), 비로소 '클래스'가 필요해진다.
* 이때는 반드시 데이터를 `private`으로 숨겨 외부의 임의 접근을 막고, 검증 로직이 포함된 메서드(`setter` 등)를 통해서만 값을 수정하도록 강제해야 한다.
* 즉, 클래스는 내부 데이터의 유효성을 보장하고 '지켜야 할 규칙(불변성)'을 책임질 때만 그 존재가 정당화된다.

<br>

## 클래스를 뚱뚱하게 만들지 말기 (모놀리스 함정 회피)
* 모든 기능을 클래스 안에 쑤셔 넣는 '모놀리스(Monolith) 함정'을 피해야 한다.
* 함수가 `private` 데이터와 직접 상호작용하여 유효성을 유지해야만 할 때만 클래스 내부의 '멤버 함수(메서드)'로 작성한다.
* 반면, 이미 존재하는 `public` 인터페이스(Getter 등)만으로 충분히 구현할 수 있는 기능이라면, 클래스 외부의 '일반 함수(비멤버 함수)'로 분리해야 한다.
* 유니티에서 플레이어의 핵심 로직(Player.cs)과 화면 출력(PlayerUI.cs)을 철저히 분리하는 것처럼, 책임을 나누어 결합도를 낮추는 것이 유지보수하기 좋은 깨끗한 코드를 만드는 핵심이다.
