웹 애플리케이션이라는 용어를 곱씹게 되었다.

웹 페이지와는 차이점이 분명히 있는데 그 차이가 명확히 떠오르지 않아서 공부가 필요할 것 같았다.

왜 '애플리케이션'이라는 용어가 굳이 붙게 되었는지, 웹 페이지와 웹 애플리케이션의 경계는 정확히 어디서 나뉘는 건지를 알아보게 되었다.  
<br>

## 웹페이지 vs 웹 애플리케이션

결론부터 말하자면 웹페이지와 웹 애플리케이션의 차이는 상호작용의 깊이에 있다.

```javascript
// 웹페이지: 정보 표시가 주목적
<div>
  <h1>회사 소개</h1>
  <p>저희 회사는...</p>
</div>

// 웹 어플리케이션: 사용자 데이터 처리와 상태 관리
const [userTasks, setUserTasks] = useState([]);
const [isLoading, setIsLoading] = useState(false);

const addTask = (newTask) => {
  setUserTasks([...userTasks, newTask]);
  // 서버에 저장하고, 상태 업데이트하고...
};
```

웹페이지는 주로 정보 전달이 목적이다. 

뉴스 사이트나 회사 홈페이지처럼 “보여주는” 것이 핵심이다. 

반면 웹 애플리케이션은 사용자와의 상호작용을 포함한 특정 작업을 수행한다. 

Gmail에서 메일을 쓰거나, Netflix에서 영상을 시청하는 것처럼 말이다.

그럼 상호작용만 있으면 모두 웹 애플리케이션일까?

처음에는 이렇게 생각했다. “버튼 클릭하고 애니메이션 나오면 웹 애플리케이션 아닌가?”  
하지만 조사해 보니 그렇지 않다는 걸 알았다.

버튼을 클릭하고 애니메이션이 나오는 것은 단순한 UI 인터랙션에 해당한다.

반면에 애플리케이션이라고 불릴만한 기능 중심의 로직 처리는 이와 다르다.

진짜 웹 애플리케이션이 되려면 비즈니스 로직, 사용자별 데이터 처리, 다중 상태 관리 같은 것들이 필요하다는 걸 깨달았다.

특히  다중 상태와 라우팅이 웹 애플리케이션의 핵심이다.  
<br>

## 다중 상태와 라우팅

그럼 여기서 새로운 궁금증이 생겼다.  
‘다중 상태와 라우팅이 뭐길래 이게 웹 애플리케이션의 핵심이라고 하는 거지?’

다중 상태는 하나의 화면에서 여러 가지 상태를 동시에 관리하는 것이다.

예를 들어 쇼핑몰에서는 동시에 이런 상태들을 관리해야 한다

- 사용자 로그인 상태
- 장바구니 아이템들
- 현재 보고 있는 페이지
- 로딩 중인지 여부
- 에러 발생 여부

이 모든 걸 한 번에 관리하면서 사용자 경험을 해치지 않아야 한다.

이러한 복잡한 상태 관리 문제를 해결하기 위해 등장한 것이 SPA(Single Page Application)이다.

원래라면 페이지를 이동할 때마다 서버에  새로운 html 파일을 요청해서 페이지를 새 로고침해야 한다.

하지만 SPA에서는 javascript로 DOM을 조작하고 필요할 경우 API를 호출하는 방식으로 매번 새로운 html을 요청하지 않아도 된다.

즉, 최초의 로드만 있다면 페이지를 새로고침 하는 일이 드물어지는 것이다.

덕분에 사용자 상태가 유지되면서도 빠른 페이지 전환이 가능해진다.  
<br>

## 웹 애플리케이션 vs 네이티브 앱

그럼 웹 애플리케이션과 네이티브 앱은 어떤 차이가 있는 걸까?

기본적으로 아래와 같은 차이점이 있다.

| 구분 | 웹 어플리케이션 | 네이티브 앱 |
| --- | --- | --- |
| 설치 | 브라우저 URL 입력 | 앱스토어 다운로드 |
| 디바이스 접근 | 브라우저 제한 내에서만 | 하드웨어 직접 접근 |
| 성능 | 브라우저 의존적 | 최적화된 네이티브 성능 |
| 개발비용 | 상대적으로 저렴 | 플랫폼별 개발 필요 |

웹 애플리케이션의 큰 장점은 크로스 플랫폼이다. 한 번 개발하면 데스크톱, 모바일, 태블릿 모든 곳에서 작동한다. 반면 네이티브 앱은 디바이스 기능을 최대한 활용할 수 있다는 게 강점이다.  
<br>

## PWA와 하이브리드

조사하다 보니 더 흥미로운 기술들을 발견했다.

웹의 편리함과 앱의 성능, 이 둘을 합친 기술인 PWA와 하이브리드 앱이다.

PWA(Progressive Web App)는 해석하자면 진보된 웹 앱이라는 의미로, 기존의 웹 애플리케이션이 더 진화한 형태이다.

웹 기술로 만들었는데 네이티브 앱처럼 동작하는 것이다.

예를 들어 브라우저에서 특정 웹 애플리케이션에 들어간 후 크롬의 '홈화면에 추가'를 하면 네이티브 앱이 설치된 것처럼 스마트폰의 바탕화면에서 해당 웹 애플리케이션으로 접근이 가능하다.

PWA는 동시에 Service Worker를 이용하는데, 이것이 백그라운드에 돌면서 캐싱, 푸시 알림, 오프라인 지원 등을 처리한다. 

마치 네이티브 앱의 백그라운드 서비스와 비슷한 역할을 하는 셈이다.  
<br>

물론 PWA를 고려하지 않고 만든 웹 페이지도 홈화면에 추가가 가능하지만, PWA를 고려한 웹 페이지보다는 여러 측면에서 불편한 점과 부족한 점이 존재한다.

그 예로 manifest 파일이 있다.

이 파일 안에는 이름, 웹에 대한 설명, 아이콘, 실행 방식, 시작 URL 등의 정보가 들어있다.

만약 manifest 파일이 없다면 다음과 같은 한계점이 존재한다.

-   앱 이름: `<title>` 태그 내용을 그대로 사용 (길고 부적절할 수 있음)
-   아이콘: favicon이나 Apple Touch Icon만 사용 (해상도 제한적)
-   실행 방식: 일반 브라우저 탭으로 열림 (네이티브 앱 같은 느낌 X)
-   시작 URL: 현재 페이지 URL 사용 (최적화되지 않음)

하이브리드 앱은 웹 기술로 개발하지만 네이티브 컨테이너에 넣어서 앱스토어에 배포하는 방식이다.

이를 이용하면 웹 개발의 편리함은 유지하면서도 네이티브 기능을 사용할 수 있다.  
<br>

## 마무리

웹은 점점 더 동적이고 효율적으로 바뀌고 있다는 느낌을 강하게 받는다.

그리고 웹의 활용 범위 또한 넓어지고 있다.

나중에는 대부분의 애플리케이션을 설치하지 않고 웹 기술을 이용하는 미래가 오지 않을까라는 생각을 해본다.
