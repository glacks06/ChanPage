compile의 사전적 의미이다. 번역하면 다른 곳들로부터 정보를 모으고 정렬하는 것을 의미한다.  
즉, 엮다 혹은 편집하다라는 뜻이다.  
컴퓨터 분야에서 compile도 이 의미에서 벗어나지 않는다.  
<br>

## Compile이란
```
 Compile is the process of converting human-readable code into machine-readable code.
 컴파일은 인간이 읽을 수 있는 코드를 기계가 읽을 수 있는 코드로 변환하는 과정이다.
```  
   
컴퓨터는 2진수를 사용한다.  
그러나 우리가 2진수로 프로그래밍을 하기에는 너무 어렵다.  
그래서 우리가 읽을 수 있는 형태로 프로그래밍을 하고, 이를 기계가 읽을 수 있는 형태로 변환하는 과정이 필요하다.  
<br>

## Compile 과정

컴파일은 전처리(Pre-processing), 컴파일(Compilation), 어셈블리(Assembly), 링킹(Linking) 과정을 거친다.

## 전처리(Pre-processing)

전처리 단계는 전처리 지시어들을 처리하는 과정이다.  
전처리 단계에서는 다음과 같은 작업을 수행한다.

-   주석 제거
-   매크로 언어 해석, 치환
-   헤더 파일 삽입

주석은 프로그램 실행과는 무관하기 때문에 이를 제거하는 과정이 전처리 과정에서 이루어진다.  
   
전처리 과정에서는 매크로 언어를 해석하고 매크로를 치환하는 작업을 하는데, 여기서 매크로란 C언어를 작성할 때 '#'으로 시작하는 코드(#define 등)를 의미한다.  
예를 들어  #define이 있다면 그에 해당하는 문자열을 만날 시 문자열과 함께 적은 내용을 치환한다.  
만약 조건에 따라 작동하는 매크로인 조건부 매크로인 경우(#if, #elif 등) 조건에 해당하지 않을 시 해당 매크로는 컴파일하지 않는다.(소스 코드에 없는 것과 동일 취급)  
   
소스코드를 작성할 때 적은 헤더파일(`#include ~`)의 경우 전처리 단계에서 해당 헤더파일 내용을 복사해 소스코드에 삽입한다.  
이때 헤더 파일에 구성된 각종 함수들의 원형은 복사하지만 구현한 내용은 포함되지 않기에 이후 링킹 과정에서 이를 연결한다.  
   
위와 같이 '#'과 함께 적는 전처리 지시어들은 컴파일러에게 소스코드를 어떻게 처리할지 알려주는 역할을 하기 때문에 컴파일 과정을 효율적으로 만들어준다.  
전처리 과정을 마치면 `~.i` 형식의 소스 코드 파일이 생성된다.  
<br>

## 컴파일(Compile)

컴파일 과정은 3가지로 이루어져 있다.  
프론트엔드(Front-end), 미들엔드(Middle-end), 백엔드(Back-end)  
 

**프론트엔드(Front-end)**

프론트엔드는 프로그래밍 언어 종속적인 부분을 처리하는 단계이다.  
쉽게말해 컴파일 언어들(c, c++, java...)을 공통된 중간표현(IR)으로 만드는 단계이다.  
이 단계는 또 다시 어휘분석, 구문분석, 의미분석, 중간 표현 생성으로 나뉜다.  
   
_※ IR: Intermediate representation (중간표현)_  
_소스 코드를 표현하기 위해 컴파일러나 가상머신에 의해 내부적으로 사용되는 데이터 구조 또는 코드_  
_(출처: 위키백과)_  
   
각 단계를 거치면서 문법을 검사하고, 코드를 트리 형태로 변환하는 과정이 시행된다.  
최종적으로 컴파일 언어들의 공통된 중간 표현인 GIMPLE 트리로 변환되어 미들엔드(Middle-end)에 넘겨진다.  
   
 
**미들엔드(Middle-end)**

   
프론트엔드에서 변환된 GIMPLE 트리를  이용해 아키텍쳐 비종속적인 최적화 과정을 거친다.  
아키텍쳐 비종속적인 최적화 과정을 거친다는 것의 의미는 CPU가 어떤 종류이건 간에 상관 없이 할 수 있는 최적화이다.  
쉽게말해, CPU 맞춤 최적화를 진행하기 전에 해볼 수 있는 최적화를 진행한다는 뜻이다.   
   
미들엔드 단계를 거치면 최종적으로 RTL(Register Transfer Lanhuage)이 생성되어 백엔드(Back-end)에 넘겨진다.  
   
_※ RTL(Register Transfer Lanhuage):_  
_레지스터 전이 언어로 고급 언어와 어셈블리 언어의 중간 형태이다._  
_디지털 회로의 동작을 기술하는 저수준 언어로, 레지스터 간의 정보 이동을 표현하는데 사용한다._  
 

**백엔드(Back-end)**  
 

미들엔드에서 넘겨받은 RTL을 이용해 아키텍쳐 종속적인 최적화 과정을 거친다.  
즉, 각 CPU 아키텍쳐에 맞는 최적화를 진행한다는 의미이다.  
   
최적화 작업이 끝나면 '~.s' 형식의 어셈블리어 파일이 생성된다.  
   
_※ 이렇게 생성된 어셈블리어 코드는 각 CPU 아키텍쳐에 맞게 생성되었기 때문에 해당 아키텍쳐에서만 실행 가능._  

_※ 어셈블리어 파일에 작성되어 있는 어셈블리 언어는 이진 형태도 아니고 고급 언어도 아닌 중간 상태이다. 기계 명령어와 실행에 필요한 다른 유용한 데이터들이 작성되어 있다._  
   

## 어셈블리(Assembly)

이전 단계인 컴파일(Compile)과정에서 생성된 어셈블리어 파일('.s' 형식)을 CPU가 읽을 수 있는 기계어(바이너리 코드)로 변환하는 과정이 어셈블리(Assembly)과정이다.  
이 변환 과정을 어셈블러(Assembler)가 수행한다.  

어셈블러의 특징으로는 다음과 같다.  
1\. CPU가 수행할 연산의 대상(데이터)를 적절히 이동하고, 수정.  
2\. 어셈블리 명령어 하나가 특정한 기계 명령어 하나와 직접 대응됨  
3\. 어셈블리어만을 처리하기 때문에 컴파일러보다 실행 속도가 더 빠르다.  
   
_※ 어셈블러의 특징 3번에 대한 부가 설명_  
_어셈블러의 특징 2번과 같이 어셈블리 명령어 하나가 기계 명령어 하나와 1:1 매칭되기 때문에 변환 과정이 단순하다._  
_반면, 컴파일러는 C, C++같은 고수준 언어를 분석해서 어셈블리어로 변환해야 하기 때문에 실행 속도가 더 느릴 수 밖에 없다._  
   
어셈블리 단계에서는 최종적으로 '~.o' 형식의 오브젝트 파일을 생성한다.  
   
※ 오브젝트 파일은 기계어 코드 또는 바이트코드와 기타 데이터 및 메타데이터를 포함하는 파일이다. 기계어나 이에 준하는 이진 코드로 이루어져있기 때문에 오브젝트 파일부터는 사람이 읽을 수 없는 형태로 작성되어진다.  
 

## 링킹(Linking)

여러 오브젝트 파일과 라이브러리를 결합해 최종 프로그램을 생성하는 단계이다.  
결과적으로 이 작업을 통해 하나의 실행파일이 만들어지는 것이다.  
링커(linker)가 해당 작업을 수행한다.  
   
링커의 역할은 다음과 같다.  
1\. 심볼 연결: 여러개의 오브젝트 파일 속에서 함수나 변수 참조가 올바르게 연결되게 한다.  
2\. 메모리 주소 할당: 오브젝트 파일에 정의된 코드와 데이터를 메모리에 적절히 배치한다.  
3\. 라이브러리 결합: 전처리 단계에서 라이브러리에 대한 내용을 복사하여 삽입했으나 여기에는 구현 내용이 들어있지 않다. 따라서 이러한 구현 내용을 찾아서 포함시킨다.  
4\. 재배치: 각 오브젝트 파일은 독립적으로 컴파일된다. 그래서 각 오브젝트 파일에 있는 데이터 또는 코드의 메모리 참조 주소가 링커에 이해 합쳐진 후와 다르기 때문에 실제 실행했을 때의 메모리 주소를 조정해야 한다. 이 과정을 재배치(Relocation)라고 한다.  
   
링킹 단계에서는 최종적으로 실행파일이 생성된다.  
 

## 마무리

컴파일은 편집하다, 엮다라는 뜻을 가진 것에 걸맞게 사람이 읽을 수 있는 고수준의 언어를 CPU가 처리할 수 있도록 기계어로 편집하고 필요한 파일들을 엮는 과정이였다.  
그 과정에서 코드를 검사하기도 하고, 자료구조 형태로 변환하기도 하는 등의 심오한 단계를 거친다.  
게다가 컴파일 과정을 조사하다보니 각 개념들마다 더 깊게 알아볼 여지가 충분해서 전부 알아보려면 시간이 꽤나 걸릴 것 같다.  
   
조사 중 구문 분석 트리나 GIMPLE 트리를 보면서 [리엑트](https://hchan-frm.tistory.com/75)를 알아보던 중 알게된 DOM이 떠올랐다. 물론 DOM이 언급되는 HTML은 프로그래밍 언어가 아니지만, 파일을 트리형태로 변환시키는 점을 보고 트리 형태가 자료를 정리하고 이용하는 데에 얼마나 중요한 것인지를 더 알아보고 싶다는 생각이 들었다. 트리라는 구조가 왜 나왔으며, 이것 말고 다른 방식은 있는지와 더 좋은 표현 방식은 없을지에 대해 고민해보는 것도 재밌을 것 같다.
